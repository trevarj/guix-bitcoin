# guix-bitcoin Phase 3 Implementation Plan — rust-bitcoin Crate Family

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the rust-bitcoin library family available from the channel: `rust-bitcoin`, `rust-bitcoin-hashes`, `rust-secp256k1`, `rust-miniscript`, `rust-bdk-wallet` — as compile-checked Guix packages whose dependency crates are hash-pinned in the channel's crate table.

**Architecture:** Extends `btc/packages/rust-crates.scm` (created in phase 2) with crate-source entries and lookup table rows for each library, generated by `guix import crate --lockfile`. A new `btc/packages/rust-bitcoin.scm` defines the five public packages with `cargo-build-system`; building one compiles the library and runs its unit tests, which is the verification value the channel adds over raw crate sources.

**Tech Stack:** Guix cargo-build-system (current crate-source/define-cargo-inputs idiom), crates.io sources.

**Spec:** `docs/superpowers/specs/2026-06-12-guix-bitcoin-channel-design.md` (Tier 2, vendored)

**Conventions:** Same as phases 1–2 (copyright header, signed commits, `-L .`, `@HASH@` discipline). **Build verification deferred** — only `guix repl` load checks now.

**Versions (verified 2026-06-12):** bitcoin **0.32.100** (0.33 still beta), bitcoin_hashes **1.0.0**, secp256k1 **0.31.1**, miniscript **13.1.0**, bdk_wallet **3.0.0**.

### Task 1: Extend the crate table

**Files:**
- Modify: `btc/packages/rust-crates.scm`

- [ ] **Step 1: Generate lockfiles and import**

For each library crate, create a workspace to produce a Cargo.lock, then
import (crates have no committed lockfile of their own; generate with a
matching toolchain via `guix shell rust rust:cargo`):

```bash
for crate in bitcoin@0.32.100 bitcoin_hashes@1.0.0 secp256k1@0.31.1 \
             miniscript@13.1.0 bdk_wallet@3.0.0; do
  name=${crate%@*}; ver=${crate#*@}
  dir=/tmp/lock-$name; mkdir -p $dir/src
  printf '[package]\nname = "lock-%s"\nversion = "0.0.0"\nedition = "2021"\n\n[dependencies]\n%s = "=%s"\n' \
         "$name" "$name" "$ver" > $dir/Cargo.toml
  echo 'fn main() {}' > $dir/src/main.rs
  (cd $dir && guix shell rust rust:cargo -- cargo generate-lockfile)
  guix import crate --lockfile=$dir/Cargo.lock "lock-$name" \
       > /tmp/crates-$name.scm
done
```

(Adjust flag spelling to the current importer; the goal is one
`crate-source` form per locked dependency. Note `bdk_wallet` is the crate
name; the Guix-side symbol is `bdk-wallet`.)

- [ ] **Step 2: Merge into `btc/packages/rust-crates.scm`**

Merge all generated `crate-source` definitions (deduplicate identical
crate+version entries across the five sets — keep one definition each) and
add lookup rows:

```scheme
(define-cargo-inputs lookup-cargo-inputs
  (electrs => (list …))                ;existing, from phase 2
  (rust-bitcoin => (list …))
  (rust-bitcoin-hashes => (list …))
  (rust-secp256k1 => (list …))
  (rust-miniscript => (list …))
  (rust-bdk-wallet => (list …)))
```

Each row lists only that library's locked dependency closure (the generated
per-app lists). The library's own crate is NOT in its row (it is the
package source, Task 2).

- [ ] **Step 3: Verify module loads**

```bash
guix repl -L . <<'EOF'
(use-modules (btc packages rust-crates))
(format #t "~a~%" (map (lambda (s) (length (lookup-cargo-inputs s)))
                       '(electrs rust-bitcoin rust-bitcoin-hashes
                         rust-secp256k1 rust-miniscript rust-bdk-wallet)))
EOF
```

Expected: six non-zero list lengths (any zero — except possibly
bitcoin_hashes, which may have zero or near-zero deps — means a lookup row
is misnamed).

- [ ] **Step 4: Commit**

```bash
git add btc/packages/rust-crates.scm
git commit -m "packages: extend crate table for rust-bitcoin family

* btc/packages/rust-crates.scm: Add locked dependency sources and
lookup rows for bitcoin, bitcoin_hashes, secp256k1, miniscript,
bdk_wallet."
```

### Task 2: `btc/packages/rust-bitcoin.scm`

**Files:**
- Create: `btc/packages/rust-bitcoin.scm`

- [ ] **Step 1: Write the module**

All five follow one pattern; here it is in full for `rust-bitcoin`, and the
table below gives the per-package values to repeat the pattern with (write
all five out fully in the real file — no abbreviations):

```scheme
;;; <copyright header from btc/packages/libraries.scm>
;;; guix-bitcoin --- Bitcoin ecosystem packages for GNU Guix
(define-module (btc packages rust-bitcoin)
  #:use-module (guix packages)
  #:use-module (guix build-system cargo)
  #:use-module ((guix licenses) #:prefix license:)
  #:use-module (btc packages rust-crates))

(define-public rust-bitcoin
  (package
    (name "rust-bitcoin")
    (version "0.32.100")
    (source (origin
              (inherit (crate-source "bitcoin" version
                                     "@HASH@"))))
    (build-system cargo-build-system)
    (arguments (list #:install-source? #t))
    (inputs (lookup-cargo-inputs 'rust-bitcoin))
    (home-page "https://github.com/rust-bitcoin/rust-bitcoin")
    (synopsis "Rust library for Bitcoin data structures and protocols")
    (description
     "This crate provides de/serialization, parsing and execution of
Bitcoin data structures and network messages: transactions, blocks,
addresses, scripts and PSBTs.")
    (license license:cc0)))
```

| package | crate | version | synopsis seed | license |
|---|---|---|---|---|
| rust-bitcoin | bitcoin | 0.32.100 | Bitcoin data structures and protocols | cc0 |
| rust-bitcoin-hashes | bitcoin_hashes | 1.0.0 | Hash types used by rust-bitcoin | cc0 |
| rust-secp256k1 | secp256k1 | 0.31.1 | Rust bindings to libsecp256k1 | cc0 |
| rust-miniscript | miniscript | 13.1.0 | Miniscript: structured Bitcoin Script | cc0 |
| rust-bdk-wallet | bdk_wallet | 3.0.0 | Bitcoin Dev Kit descriptor wallet library | expat/apache2 dual |

Implementation notes:
- `crate-source` (from `(guix build-system cargo)`) already returns an
  origin; if `(origin (inherit …))` is redundant in current Guix, use the
  `crate-source` call directly as the `source` value.
- Hashes: `guix download $(guix repl …crate-uri…)` is awkward — simplest is
  `guix download "https://crates.io/api/v1/crates/<crate>/<version>/download"`
  which prints the base32 hash (crate tarballs are stable, unlike GitHub
  archives).
- `rust-secp256k1`: add `(inputs (cons (specification->package "libsecp256k1") …))`?
  NO — the crate vendors its own C code by default; keep it self-contained.
- Licenses: verify each crate's actual license field from its Cargo.toml at
  the pinned version and correct the table if it differs.
